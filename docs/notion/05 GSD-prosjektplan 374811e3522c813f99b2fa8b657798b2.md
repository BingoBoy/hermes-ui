# 05 GSD-prosjektplan

# Formål

Denne siden er GSD Master-planen for utviklingen av Hermes UI for Bob.

# GSD-flyt

Prosjektet følger:

```
discuss → plan → execute → verify
```

med små planer, atomiske oppgaver og tydelig verifisering.

# Startkommandoer i Cursor

```
/gsd-new-project
/gsd-map-codebase
/gsd-discuss-phase
/gsd-plan-phase
/gsd-execute-phase
/gsd-verify-work
```

# Fase 1 – Discuss

```xml
<phase id="01" name="Discuss">
  <task id="01.1">
    <title>Kartlegg Hermes på Bob</title>
    <context>Eksisterende terminalworkflow, Bob, Hermes, launchctl, Cloudflare SSH</context>
    <output>Dokumentert liste over kommandoer, paths, tjenester og loggfiler</output>
  </task>
  <task id="01.2">
    <title>Avklar MVP-scope</title>
    <context>Terminalgrensesnittet oppleves begrensende</context>
    <output>5–8 godkjente UI-handlinger for første versjon</output>
  </task>
</phase>
```

# Fase 2 – Plan

```xml
<phase id="02" name="Plan">
  <task id="02.1">
    <title>Velg teknisk grunnstruktur</title>
    <context>Flask/FastAPI, enkel frontend, Cloudflare Access</context>
    <output>Besluttet stack og repo-struktur</output>
  </task>
  <task id="02.2">
    <title>Definer API-kontrakt</title>
    <context>Status, logs, start, stop, restart</context>
    <output>Minimal API-spesifikasjon</output>
  </task>
  <task id="02.3">
    <title>Definer sikkerhetsmodell</title>
    <context>Ingen fri shell, allowlist, Cloudflare Access</context>
    <output>Sikkerhetsregler for MVP</output>
  </task>
</phase>
```

# Fase 3 – Execute

```xml
<phase id="03" name="Execute">
  <task id="03.1">
    <title>Opprett repo og grunnstruktur</title>
    <context>hermes-ui, README, docs, backend</context>
    <output>Initielt repo med ren struktur</output>
  </task>
  <task id="03.2">
    <title>Bygg status-backend</title>
    <context>GET /api/status og GET /api/hermes/status</context>
    <output>Fungerende lokal backend</output>
  </task>
  <task id="03.3">
    <title>Bygg første dashboard</title>
    <context>Statuskort, Hermes-status, loggvisning</context>
    <output>Enkelt webgrensesnitt lokalt på Bob</output>
  </task>
</phase>
```

# Fase 4 – Verify

```xml
<phase id="04" name="Verify">
  <task id="04.1">
    <title>Verifiser lokal drift</title>
    <context>127.0.0.1 på Bob</context>
    <output>Backend og UI fungerer lokalt</output>
  </task>
  <task id="04.2">
    <title>Verifiser Cloudflare-tilgang</title>
    <context>Cloudflare Tunnel og Cloudflare Access</context>
    <output>UI kan nås eksternt etter innlogging</output>
  </task>
  <task id="04.3">
    <title>Verifiser sikkerhetsgrenser</title>
    <context>Ingen fri shell, ingen secrets i logger</context>
    <output>Bekreftet sikker MVP</output>
  </task>
</phase>
```

# Commit-prinsipp

Bruk atomiske commits:

```bash

git commit -m "docs: add hermes ui mvp spec"
```

```bash
git add .
git commit -m "feat: add status api endpoint"
```

```bash
git add .

```

# Verifisering før ship

Før hver ship:

```bash
git status
```

```bash
pytest
```

eller relevant testkommando når prosjektet er opprettet.
